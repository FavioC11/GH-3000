Objetivos del módulo
  • Describir cómo VS Code, el SDK de .NET y el C# Dev Kit admiten pruebas unitarias en C#.
  • Usar el modo Agente en Copilot Chat para generar pruebas unitarias con /setupTests y /tests.
  • Usar los agentes Plan y Agent para diseñar estrategias y automatizar flujos de prueba de varios archivos.
  • Ampliar cobertura con texto fantasma y corregir fallos con el Test Explorer y /fixTestFailure.

Pruebas unitarias con GitHub Copilot

Setup → Generar → Ejecutar → Mantener: un flujo completo sin salir de Visual Studio Code.

Entorno de pruebas — mnemónico: "XNM: eXcelentes Números Miden"
Explorador de pruebas

Vista árbol + botones de reproducción

↩ funciones

Vista árbol de todos los casos de prueba del workspace.

Botones de reproducción verde junto a cada clase y método.

Resultados en decoraciones del editor + seguimiento de pila.

Comandos en paleta: Test: Run All Tests, Debug Failed Tests, Show Output.

3 marcos — XNM

xUnit · NUnit · MSTest

↩ paquetes de cada uno

Microsoft.NET.Test.Sdk + xUnit + xunit.runner.visualstudio + coverlet

N — NUnit:

Microsoft.NET.Test.Sdk + NUnit + NUnit3TestAdapter

M — MSTest:

Microsoft.NET.Test.Sdk + MSTest.TestAdapter + MSTest.TestFramework + coverlet

3 fases del flujo

SGE: Setup, Generar, Ejecutar

↩ detalle

S — Setup: VS Code + .NET SDK + C# Dev Kit + proyecto de prueba referenciado.

G — Generar: Copilot Chat (Ask/Plan/Agent) → /setupTests, /tests, lenguaje natural.

E — Ejecutar/Mantener: Test Explorer + texto fantasma + /fixTestFailure.

3 Agentes en la vista Chat — mnemónico: "APA: Analizo, Planifico, Actúo"
A — Ask (Preguntar)

Solo lectura, sin modificar archivos

↩ cuándo usarlo

Explora casos perimetrales antes de escribir código.

Compara marcos de prueba o estilos de aserción.

Ve ejemplos de prueba en el chat sin escribirlos al disco.

Ideal para análisis previo antes de pasar a Plan o Agent.

P — Plan

Estrategia revisable antes de codificar

↩ flujo

1. Abre los archivos a probar.

2. Selecciona Plan y describe el trabajo.

3. El agente hace preguntas aclarantes (marco, convenciones).

4. Genera plan con resumen, pasos y verificación.

5. Itera el plan hasta aprobarlo y envíalo al Agente.

Ideal para varias archivos o convenciones de equipo.

A — Agent (Agente)

Flujo autónomo de varios archivos

↩ flujo

Edita archivos, ejecuta comandos en terminal, reintenta tras errores.

Scaffolding del proyecto, genera y ejecuta pruebas en un solo prompt.

Confirma o rechaza cada invocación de herramienta.

Revisa diffs antes de aceptar con Mantener/Deshacer.

Cada transferencia ~1 PRU; tarea compleja puede ser 3–5 PRUs.

Comandos para generar pruebas — en la vista Chat con el Agente
/setupTests

Configura el marco de pruebas

↩ detalle

Recomienda un marco y guía la instalación.

En modo Agente: instala paquetes y crea el proyecto automáticamente.

Útil al iniciar un nuevo proyecto o al incorporar uno sin pruebas.

/tests (archivo)

Todo el archivo activo en el editor

↩ flujo

Abre el archivo a probar → Chat con Agente → escribe /tests + instrucción.

Copilot detecta el marco existente y el estilo de codificación.

Anexa a archivo existente o crea uno nuevo en la ubicación adecuada.

Confirma cada herramienta, revisa diff → Mantener o Deshacer.

/tests (selección)

Método o bloque seleccionado

↩ flujo

Selecciona el método en el editor → Chat con Agente.

Escribe: /tests Generate tests for the selected method. Include success, failure, and edge cases.

Revisa y acepta o descarta los cambios resultantes.

Lenguaje natural

Sin comandos de barra diagonal

↩ ejemplos

"Genera pruebas xUnit para los métodos y agrégalas al proyecto Calculator.Tests".

"Escribe pruebas para CalculateDiscount con casos límite (negativos, cero). Ejecuta las pruebas después".

"Crea pruebas de integración para la capa de acceso a datos".

Contexto y personalización — mnemónico: "BAME: Botón, Arrastra, Menciona, Externo"
4 formas de adjuntar contexto

BAME

↩ detalle

B — Botón Agregar contexto: archivos, carpetas, símbolos, selección del editor.

A — Arrastra archivos desde el Explorador o pestañas al Chat.

M — #Menciones: #selection, #editor, #codebase, nombre de archivo/símbolo.

E — Externos: abre archivos MD (convenciones) y adjúntalos via Agregar contexto.

Instrucciones personalizadas

*.instructions.md en el workspace

↩ qué controlan

Marco preferido (xUnit vs NUnit).

Convenciones de nomenclatura de clases y métodos de prueba.

Estructura: Arrange-Act-Assert.

Patrones: pruebas parametrizadas para valores límite.

Usa applyTo: tests/** para limitar al directorio de pruebas. Compartir en control de código fuente.

Extensión y corrección de pruebas — mnemónico: "FEFA: Fantasma, Explorer, Fix, Agente"
F — Texto fantasma

Casos adicionales sin salir del editor

↩ cuándo funciona mejor

Cursor al final del último test → Enter → escribe un comentario descriptivo o el inicio del método.

Copilot sugiere el test completo con Tab para aceptar.

Funciona mejor cuando: ya hay patrones AAA, el método está importado, el comentario describe el escenario.

Para trabajo mayor (clase nueva), usa la vista Chat.

E — Explorer (sparkle)

Un clic desde el Test Explorer

↩ flujo

Ejecuta la prueba → falla → hover sobre prueba con error.

Clic en botón Corregir error de prueba (ícono sparkle ✨).

Copilot abre chat con la prueba fallida + salida como contexto y propone corrección.

Aplica con Conservar o descarta con Deshacer → reejecutar para confirmar.

F — /fixTestFailure

Desde la vista Chat con contexto extra

↩ cuándo usarlo

Abre Chat → escribe /fixTestFailure.

Adjunta contexto extra: archivos relacionados o salida del terminal.

Ideal cuando hay varios fallos a la vez o necesitas adjuntar más contexto que el sparkle.

Sigue sugerencias → reejecutar para confirmar.

A — Agente automático

Ciclo ejecutar-corregir-reejecutar

↩ prompt ejemplo

Selecciona Agente → prompt:

"Run the xUnit tests in Calculator.Tests. If any fail, propose and apply fixes, then rerun until they pass."

El Agente supervisa la salida, identifica errores y repara en varios archivos automáticamente.

Confirma cada invocación de herramienta; revisa cambios antes de aceptar.

¿Qué herramienta usar? — guía rápida de decisión
Generar pruebas

¿Qué agente elijo?

↩ criterios

Ask → análisis previo, sin modificar archivos.

Plan → varios archivos, ambigüedad o convenciones del equipo por confirmar.

Agent → tarea bien definida, quieres scaffolding + generación + ejecución en un solo paso.

Plan → Agent → máximo control: plan revisable + implementación autónoma.

Corregir / ampliar

¿Qué herramienta elijo?

↩ criterios

Texto fantasma → agregar casos a un archivo de prueba existente con patrón visible.

Sparkle (Explorer) → un solo fallo, corrección rápida y específica.

/fixTestFailure → varios fallos o necesitas adjuntar más contexto.

Agente automático → ejecutar + corregir + reintentar en varios archivos en una sesión.

Resumen

Entorno XNM: VS Code + .NET SDK + C# Dev Kit con Explorador de pruebas. Marcos: xUnit, NUnit, MSTest.
Agentes APA: Ask (analiza sin modificar), Plan (estrategia revisable), Agent (flujo autónomo multi-archivo). Decide: solo→Agent, ambiguo→Plan+Agent.
Comandos: /setupTests para configurar, /tests para generar (archivo completo o selección), lenguaje natural con contexto BAME (#selection, #codebase, instrucciones .md).
Extensión y corrección FEFA: Fantasma (casos adicionales en editor), Explorer sparkle (1 fallo rápido), /fixTestFailure (varios fallos + contexto), Agente automático (ciclo ejecutar-corregir-reejecutar).
