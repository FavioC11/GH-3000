Servidor MCP de GitHub

El "USB-C para herramientas de IA": conecta GitHub Copilot a herramientas y datos externos con un estándar abierto, sin configuración compleja.

¿Qué es MCP? — mnemónico: "LPR: Local, Puente, Remota"
MCP en una línea

El USB-C para IA

↩ definición

Estándar abierto (Anthropic) para que los modelos de IA se conecten a herramientas y datos.

✅ Biblioteca creciente de herramientas listas para usar.

✅ Flexibilidad: funciona con distintos proveedores de IA.

✅ Se integra en entornos y procesos existentes.

L — Local ↔ Local

Cliente MCP + servidor local + datos locales

↩ cuándo usarlo

El cliente MCP (IDE, Claude, etc.) se comunica con un servidor local.

El servidor accede a archivos, bases de datos u otros recursos locales.

Ideal para: desarrollo local con datos privados que no deben salir del equipo.

P — Puente / R — Remota

Servidor local como puente · Todo remoto

↩ cuándo usarlos

Puente: servidor local que llama APIs web remotas. Útil para caché, seguridad o preprocesado antes de pasar a servicio remoto.

Remota: cliente → servidor en Internet → APIs externas. Para servicios SaaS, plataformas cloud o integraciones de terceros que solo existen online.

Servidor MCP de GitHub — mnemónico: "SOWSE: Sin Docker, OAuth, Web/Móvil, SSO, Escala"
Qué hace

+30 herramientas disponibles

↩ capacidades

Analizar y resumir código para entender mejor los proyectos.

Crear y administrar issues y pull requests.

Automatizar triage del repositorio y seguimiento de tareas.

Agregar issues, editar archivos, crear ramas.

Clasificar PRs e issues por prioridad.

Ventajas SOWSE

Por qué usar el servidor hospedado

↩ ventajas

S — Sin Docker: no requiere configuración manual ni archivos docker.

O — OAuth: login de un clic para autenticación rápida.

W — Web/Móvil: funciona en entornos web, desktop y móvil.

S — SSO: compatible con Entra y Auth0 para empresas.

E — Escala: escala automáticamente según el uso.

vs. servidor local

Hospedado en GitHub vs. Docker local

↩ comparativa

Local (Docker): requiere Docker, gestión de tokens y config manual. No funciona con clientes web como GitHub.com.

Hospedado GitHub: conexión rápida, OAuth, funciona en web/móvil/desktop, escala automática, SSO empresarial.

GitHub Enterprise Server con restricciones de PAT puede limitar la disponibilidad.

3 formas de configurar en VS Code — mnemónico: "OPD: OAuth, PAT, Docker"
O — OAuth

Recomendado · Más sencillo

↩ pasos

1. Paleta de comandos: Ctrl+Shift+P → MCP: Add Server.

2. Seleccionar HTTP (HTTP o Server-Sent Events).

3. URL: https://api.githubcopilot.com/mcp/ → Enter.

4. ID del servidor: Enter (predeterminado) o uno personalizado.

5. Elegir ámbito: usuario (todos los proyectos) o workspace (proyecto actual).

6. Autorizar con GitHub → Permitir.

P — PAT

Control avanzado de permisos

↩ pasos adicionales

1. Crear PAT con scope repo y read:packages en GitHub.

2. Seguir pasos de OAuth pero cancelar el OAuth cuando se solicite.

3. Añadir en config:

"headers": {"Authorization": "Bearer ${input:github_token}"}

4. Añadir entrada de contraseña segura:

"inputs": [{"id": "github_token", "type": "promptString", "password": true}]

5. Reiniciar servidor MCP → escribir PAT cuando se solicite.

D — Docker

Local · Opcional · GHE Server

↩ requisitos y config

Requisitos: Docker instalado y corriendo + PAT (OAuth no soportado).

Imagen: ghcr.io/github/github-mcp-server

Config con command: docker, args run -i --rm -e GITHUB_PERSONAL_ACCESS_TOKEN.

Útil para GitHub Enterprise Server con restricciones de PAT o políticas de red estrictas.

Modo Agente + MCP — mnemónico: "IDA: Independiente, Decide, Adapta"
Capacidades agénticas IDA

Qué aporta el modo agente

↩ capacidades

I — Independiente: flujos de trabajo de varios pasos sin instrucciones constantes.

D — Decide: elige qué herramientas o enfoques usar según el contexto disponible.

A — Adapta: responde a comentarios, ajusta el enfoque e itera en los resultados.

Bucles agente con MCP

MCP potencia al agente más allá del IDE

↩ cómo funciona

MCP da a Copilot acceso a datos externos, APIs y herramientas empresariales.

Mantiene contexto en múltiples plataformas sin cambiar de app.

Bucle agente: busca información dinámicamente → analiza resultados → ejecuta siguientes pasos → itera, todo sin reiniciar desde cero.

3 ventajas combinadas

Agente + MCP juntos

↩ beneficios

Contexto extendido: extrae información de múltiples sistemas, no solo el editor.

Esfuerzo reducido: issues, flujos de trabajo y comprobaciones automatizados.

Integración sin problemas: tareas que abarcan herramientas y plataformas sin conectores personalizados.

Procedimientos recomendados y solución de problemas
5 procedimientos recomendados

Para guiar bien al agente con MCP

↩ lista

1. Objetivos claros: define el resultado final esperado.

2. Proporcionar contexto: vínculos, referencias y pasos anteriores relevantes.

3. Establecer límites: especifica si quieres que solo planee (sin cambios) y qué herramientas MCP estarán activas.

4. Pedir confirmación: haz que Copilot resuma el plan antes de cambios grandes.

5. Archivos de instrucción: crea archivos de aviso personalizados para comportamiento coherente con servidores MCP específicos.

Solución de problemas

5 verificaciones prácticas

↩ checklist

✅ Confirma que iniciaste sesión en GitHub en VS Code.

✅ Si usas PAT: verifica que tiene los ámbitos correctos y está bien escrito.

✅ Revisa la config por errores tipográficos o campos faltantes.

✅ Si usas Docker: confirma que está instalado y corriendo.

✅ Reinicia VS Code o el servidor MCP para resolver problemas de conexión temporales.

Resumen

MCP LPR: estándar USB-C para IA. 3 topologías: Local↔Local (datos privados), Puente local→remoto (caché/seguridad), Remota→remota (SaaS/cloud).
Servidor GitHub SOWSE: +30 herramientas. Sin Docker, OAuth un clic, Web/Móvil/Desktop, SSO empresarial, Escala automática.
Config OPD: OAuth (recomendado, 6 pasos), PAT (control avanzado, cancelar OAuth + headers), Docker (local, GHE Server, sin OAuth).
Agente + MCP IDA: Independiente (multi-paso), Decide (elige herramientas), Adapta (itera). Resultado: bucles agente, contexto extendido, esfuerzo reducido.
Mejores prácticas: objetivos claros, contexto, límites explícitos, confirmación previa a cambios grandes, archivos de instrucción por servidor MCP.
